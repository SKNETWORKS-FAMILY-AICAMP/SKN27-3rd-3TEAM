## 1. 메인 홈 (`app.py` / `landing/`)

### 주요 구현 내용
- Streamlit 앱 진입점(`app.py`)과 랜딩 전용 모듈(`landing/styles.py`, `landing/sections.py`) 설계
- 각 기능 섹션(도감·배틀·챗봇·팀빌더·로그인·미니게임·피피고)을 **풀스크린 슬라이드** 형태로 구성
- CSS `scroll-snap-type: y mandatory` + JavaScript `IntersectionObserver`로 자연스러운 슬라이드 전환 구현
- 장식용 포켓몬 스프라이트를 클릭하면 랜덤 포켓몬 GIF로 교체되는 인터랙션 추가

### 핵심 기술 포인트
- Streamlit의 `st.markdown(unsafe_allow_html=True)`으로 HTML/CSS/JS를 직접 주입
- `iframe` 내부에서 `window.parent.document`를 참조해 Streamlit 컨테이너 스타일을 동적으로 제어
- 배경 이미지·캐릭터 이미지를 Base64로 인코딩해 외부 CDN 없이 번들링(`utils/images.py`)

## 2. 로그인 (`pages/login.py` / `login/`)

### 주요 구현 내용
- **GitHub OAuth 2.0 Authorization Code Flow** 기반 소셜 로그인 구현 (별도 회원가입 없음)
- OAuth 콜백 코드 수신 → Access Token 교환 → GitHub API로 사용자 통계 수집 → 백엔드 DB Upsert 전 과정 처리
- `st.session_state` + `streamlit-cookies-controller`로 **세션 이중 영속화** (새로고침·탭 전환 후에도 로그인 유지)
- 포케볼 SVG 스핀 로딩 화면, 로그아웃 확인 다이얼로그 등 상태별 UI 분기 처리

### 핵심 기술 포인트
- `ThreadPoolExecutor(max_workers=2)`로 커밋 수·스타 수 GitHub API 병렬 수집 → 로딩 시간 단축
- DB 저장 실패 시에도 GitHub 통계로 세션을 구성해 서비스 장애를 무중단으로 처리 (Fallback)
- `scope: user:email` 최소 권한 원칙 적용

### 아쉬웠던 점
- Streamlit은 페이지 전환 시 전체 스크립트를 재실행하기 때문에 OAuth 콜백 `?code=` 파라미터를 올바른 타이밍에 감지하는 것이 까다로웠음
- GitHub OAuth 특성상 팝업이 아닌 새 탭으로 이동 후 리다이렉트되는 구조여서 `st.switch_page`와 쿠키 타이밍 문제를 여러 번 수정해야 했음
- `streamlit-cookies-controller`가 Streamlit 렌더 사이클과 맞지 않아 쿠키 복구 시점을 맞추는 데 시간이 걸렸음
- **배포 환경 로그인 차단 문제**: 배포된 환경에서 깃허브 로그인을 시도할 때, Streamlit의 iframe 샌드박스 정책과 깃허브의 보안 정책(X-Frame-Options)이 충돌하여 창이 차단되는 오류가 발생함. 이를 해결하기 위해 로그인 버튼이 부모 프레임을 벗어나 새 탭(`target="_blank"`)에서 열리도록 수정하여 보안 정책을 우회하고 문제를 성공적으로 해결함.

## 3. 마이페이지 (`pages/mypage.py` / `mypage/`)

### 주요 구현 내용
- GitHub 개발자 통계 + 미니게임 전적 + 배지 시스템 + 포켓몬 컬렉션을 통합한 **트레이너 대시보드**
- 커밋 수·레포 수 기반으로 4단계 **트레이너 등급**(Rookie → Legendary Developer) 자동 산출
- 관동지방 8개 체육관 배지를 활동 미션으로 구현한 **게이미피케이션 시스템** (마이페이지 방문, 퀴즈 도전, 팀 분석 등)
- 배틀 배지 케이스 (관장 배지 8종) + 포켓몬 컬렉션 그리드 + 최근 활동 로그 + 팀빌더 히스토리 섹션 구성

### 핵심 기술 포인트
- `ThreadPoolExecutor(max_workers=5)`로 GitHub API, 게임 통계, 게임 로그, 팀 히스토리, 챗봇 세션을 **동시 조회**해 페이지 로딩 최소화
- 세션 내 GitHub 통계 캐시(`st.session_state[f"_gh_{username}"]`)로 불필요한 API 재호출 방지
- Glassmorphism 디자인 시스템(`backdrop-filter: blur`) + 섹션별 고유 색상 보더 카드 레이아웃

### 아쉬웠던 점
- 여러 API를 병렬로 호출하면서 각 결과가 `None`일 때의 예외 처리를 빠짐없이 해야 했음
- 팀 히스토리 카드에서 Streamlit 버튼(`st.button`)과 커스텀 HTML 카드를 같은 행에 배치하려고 `st.columns`와 HTML을 혼용하는 구조가 렌더링 순서 문제를 일으켜 레이아웃이 어긋나는 경우가 있었음

## 4. 미니게임 허브 (`pages/mini_game.py` / `minigame/`)

### 주요 구현 내용
- **실루엣 퀴즈**와 **메모리 게임** 두 미니게임으로 이동하는 선택 허브 페이지
- 각 게임 카드에 캐릭터 이미지·제목·설명을 배치하고 버튼 클릭 시 해당 게임 페이지로 라우팅
- 전체 미니게임 영역의 공통 스타일(`minigame/styles.py`)과 UI 컴포넌트(`minigame/ui.py`) 모듈 분리

## 5. 포켓몬 도감 (`pages/pokedex.py` / `pokedex/`)

### 주요 구현 내용
- 전국 1,025종 포켓몬을 검색·필터링하는 **풀 기능 도감** 구현
- 이름/설명/특성 키워드 검색, 타입 다중 선택(18종), 지방 필터(10지방), 도감번호 범위 슬라이더, 특성 셀렉트박스 조합 지원
- **무한 스크롤**: JavaScript `IntersectionObserver`가 "더 보기" 버튼을 자동 클릭하도록 구현해 UX 개선
- 포켓몬 카드 클릭 시 상세 페이지(`pages/pokemon_detail.py`)로 이동

### 핵심 기술 포인트
- 타입 아이콘 SVG를 동적으로 로드해 각 타입 버튼에 인라인으로 삽입
- 무한 스크롤에서 Streamlit 내부 iframe 경계를 넘어 `window.parent.document`의 버튼을 감지·클릭하는 방식으로 처리

### 아쉬웠던 점
- Streamlit의 `st.button`은 HTML과 별개 레이어에 렌더링되어, 커스텀 CSS로 타입 아이콘 위에 투명 버튼을 겹치는 방식으로 클릭 이벤트를 처리해야 했음
- 무한 스크롤 구현 시 Streamlit의 iframe 구조 때문에 `IntersectionObserver` root를 정확히 찾는 과정에서 시행착오가 있었음
- **대량의 데이터 렌더링 성능 이슈**: 초기에는 약 2,000마리에 달하는 포켓몬 데이터를 한 번에 렌더링하려고 시도하여 브라우저에 과부하가 걸리고 렌더링 속도가 심각하게 저하되는 문제가 발생함. 이를 해결하기 위해 데이터를 한 번에 **50마리씩 끊어서 로드하는 페이지네이션(청킹) 기법**을 적용하여 초기 로딩 속도와 스크롤 성능을 대폭 개선함.

## 6. 피피고 확장 프로그램 (`pokemon_pipigo/`)

### 주요 구현 내용
- Chrome 브라우저에 설치하는 **Manifest v3 확장 프로그램**
- 선택한 포켓몬 GIF 캐릭터(피카츄·야돈·리자몽·꼬부기)가 모든 웹 페이지 위를 자유롭게 돌아다님
- 포켓몬 클릭 시 말풍선이 열려 텍스트 입력 → **Groq AI(LLaMA 3.1 8B)**가 한↔영 번역 또는 질의응답 반환
- 드래그 앤 드롭, 위치 고정 모드, 파트너 변경, 전원 토글 기능 지원
- `chrome.storage.local`로 설정 영구 저장 (재시작 후에도 유지)

### 핵심 기술 포인트
- `background.js`(서비스 워커)에서 Groq API를 호출하고 결과를 `content.js`로 메시지 패싱
- `requestAnimationFrame` 기반 60fps 물리 루프로 포켓몬 이동 처리 (랜덤 목표 좌표 + 화면 경계 클램핑)
- GIF 재생 정지 문제를 막기 위해 8초마다 `src`에 타임스탬프 쿼리를 추가해 강제 리프레시

### 아쉬웠던 점
- Chrome Manifest v3에서는 서비스 워커가 외부 API를 직접 호출하고 콘텐츠 스크립트와 메시지로만 통신해야 해서 구조 설계에 시간이 걸렸음
- 포켓몬이 모든 사이트의 DOM 위에 올라가야 하므로 사이트별 CSS 충돌을 최소화하기 위해 Shadow DOM 경계를 고려해야 했음
- API 키를 소스 코드에 평문 저장해야 하는 확장 프로그램 특성상 팀 공유 시 키 노출 문제 관리가 필요했음

## 7. 데이터 파이프라인 및 클라우드 배포 (Data & DevOps)

### 주요 구현 내용
- **데이터 수집 파이프라인 구축**: **PokeAPI**를 활용하여 포켓몬의 기본 정보, 스탯, 타입, 진화 트리 등 방대한 원시 데이터를 자동으로 수집하고 가공하는 파이프라인 스크립트를 직접 개발함
- **Railway 기반 배포**: 로컬 환경에서 완성된 프론트엔드 및 백엔드 서비스를 **Railway** 클라우드 플랫폼을 활용해 성공적으로 배포 및 호스팅함

### 핵심 기술 포인트
- 수집한 포켓몬 데이터를 서비스 요구사항에 맞춰 최적의 JSON/DB 스키마 형태로 정제 및 전처리
- Docker 컨테이너 설정을 최적화하여 Railway 배포 시 환경 변수와 포트 바인딩이 원활하게 이루어지도록 구성

## 전체 회고

프론트엔드 전반의 사용자 경험(로그인부터 마이페이지, 도감, 미니게임 허브, 메인 랜딩)을 주로 담당하면서 **Streamlit의 렌더링 구조(iframe, 재실행 사이클)**와 싸우는 경험이 가장 많았습니다. 순수 웹 개발과 달리 Streamlit은 Python 스크립트가 매 인터랙션마다 재실행되어 상태 관리와 HTML/JS 주입을 신중하게 설계해야 했습니다. 대량의 데이터를 다룰 때 50개씩 청킹하여 렌더링 속도를 최적화하거나, 배포 환경에서 샌드박스 정책으로 인해 발생하는 로그인 차단 에러를 `target="_blank"` 우회 기법으로 해결하는 등 실제 프로덕션 수준의 트러블슈팅 경험을 쌓을 수 있었습니다.

또한 피피고 확장 프로그램을 통해 Manifest v3의 서비스 워커 모델을 처음 접하며 비동기 메시지 패싱 구조를 이해했고, PokeAPI 데이터 수집 파이프라인 구축부터 Railway를 활용한 최종 서비스 배포까지 데이터-프론트-인프라를 아우르는 전반적인 사이클을 경험해 볼 수 있어 매우 뜻깊은 프로젝트였습니다.
