# 프로젝트 회고록 (Retrospective)

**프로젝트명:** 포켓몬 AI 챗봇 (Pokémon AI Chatbot)  
**작성일:** 2025-05-14  
**작성자:** -  
**담당 기능:** 포켓몬 챗봇 백엔드 (Agent, VectorStore, GraphDB, ChatHistory)

---

## 1. 프로젝트 개요

포켓몬스터 도메인 지식을 기반으로 LLM을 활용한 지능형 챗봇 애플리케이션을 개발했다.  
사용자가 자연어로 포켓몬 정보를 질의하면, 여러 데이터 소스(PostgreSQL, VectorStore, GraphDB)를 통합한 Agent가 정확한 답변을 생성한다.

---

## 2. 담당 역할 및 작업

### 2.1 담당 모듈

| 파일 | 역할 |
|------|------|
| `ingest.py` | `flavor_text.json`을 PGVector VectorStore에 임베딩하여 인덱싱 |
| `chat_history.py` | 사용자 채팅 세션 및 메시지를 PostgreSQL DB에 저장·조회 |
| `pokemon_agent.py` | PGVector 기반 VectorStore 구성, LangGraph Agent 설계 및 Tool 구현 |
| `pokemon_neo4j.py` | Neo4j GraphDB 연동, 진화 체인·타입 상성·포켓몬 약점/저항/면역 Tool 구현 |

### 2.2 주요 작업 상세

**`ingest.py` — 임베딩 파이프라인**
- `flavor_text.json`을 읽어 `"species_id: version_name content"` 형식의 `Document` 객체로 변환
- OpenAI Embeddings API를 통해 벡터화하여 `langchain_pg_embedding` 컬렉션에 적재
- 이미 데이터가 있을 경우 `similarity_search`로 사전 확인 후 중복 인덱싱 건너뜀 (idempotent)

**`chat_history.py` — 채팅 이력 관리**
- `chat_sessions` / `chat_messages` 테이블 설계 및 CRUD 구현
- 세션별 멀티턴 대화 저장, 사용된 툴 목록(`used_tools`) 함께 기록
- `ON DELETE CASCADE`로 세션 삭제 시 메시지 자동 삭제 처리

**`pokemon_agent.py` — Agent 및 VectorStore**
- PGVector VectorStore 구성 및 MMR Retriever 설정 (k=20, fetch_k=50, lambda_mult=0.7)
- BM25(`rank_bm25`) + pg_trgm + Vector 세 채널을 RRF(k=60)로 결합한 Hybrid Search 구현
- BM25 인덱스는 모듈 로드 시 1회 빌드하여 메모리에 상주, 빌드 실패 시 나머지 채널로 graceful degradation
- Query Rewriting(`_rewrite_query`) — 첫 번째 툴 호출 전 사용자 질문을 핵심 키워드 중심으로 재작성
- LangGraph 3-노드 구조 설계: `agent → tools → crag → agent`
  - `agent_node`: SystemMessage 삽입, Query Rewriting, LLM 추론, MAX 5회 제한
  - `tools_node_wrapper`: ToolNode 실행 후 `tool_call_count` 증가
  - `crag_check_node`: 툴 결과 관련성 LLM 평가, 낮으면 재검색 유도 SystemMessage 삽입
- `astream_chat` 비동기 스트리밍 구현 — `astream_events v2` 사용, `agent` 노드 토큰만 전달, rewriter 태그 필터링
- System Prompt에 오박사 페르소나 적용 (친근한 말투, 이모지 활용, 시스템 메타데이터 출력 차단)
- `search_pokemon_db`, `search_flavor_text` Tool 구현

**`pokemon_neo4j.py` — GraphDB 연동**
- Neo4j Bolt 드라이버로 진화 체인 탐색 (`search_evolution_chain`) — `LIMIT 1` + `DISTINCT`로 중복 폼 방지
- 타입 상성 조회 (`search_type_relations`) — `AGAINST` 관계 기반, 공격·방어 양방향 + 포켓몬 예시 반환
- 포켓몬 개체별 약점/저항/면역 조회 (`search_pokemon_weakness`) — 듀얼 타입 복합 배율(4배/0.25배 등) 정확 반영
- Neo4j 미수록 진화 조건 보완을 위한 `KNOWN_EVO_CONDITIONS` fallback 맵 작성 (친밀도 진화 등)

## 3. 잘 된 점 (Keep)

**VectorDB + GraphDB 통합 활용**  
VectorStore(PGVector)와 GraphDB(Neo4j)를 함께 활용하여 챗봇이 다양한 포켓몬 도메인 지식에 효과적으로 답변할 수 있도록 구성했다. 도감 설명 같은 비정형 데이터는 Hybrid Search로, 진화·상성 같은 관계형 데이터는 그래프 탐색으로 각각의 강점을 살려 처리했다.

**RAG 구조 고도화**  
단순 벡터 검색을 넘어 BM25 + pg_trgm + Vector를 RRF로 결합한 Hybrid Search와 CRAG(Corrective RAG)를 도입하여 검색 품질과 답변 정확도를 높였다. 특히 CRAG가 관련 없는 결과를 걸러내고 재검색을 유도하는 구조가 실질적인 답변 품질 향상으로 이어졌다.

**LangGraph 3-노드 설계**  
`agent → tools → crag` 흐름으로 각 노드의 역할을 명확히 분리했다. Query Rewriting으로 검색 품질을 사전에 높이고, CRAG로 사후 보정하는 구조 덕분에 복합 질문도 안정적으로 처리할 수 있었다.

**스트리밍 응답 구현**  
`astream_events v2`를 활용한 비동기 스트리밍으로 사용자가 답변을 실시간으로 확인할 수 있게 했다. rewriter 태그 필터링으로 내부 Query Rewriting 토큰이 사용자에게 노출되지 않도록 처리한 점도 잘 된 부분이다.

**오박사 페르소나**  
System Prompt에 오박사 캐릭터를 적용하여 단순한 정보 제공을 넘어 사용자 경험을 풍부하게 만들었다. 이모지 활용과 친근한 말투가 포켓몬이라는 도메인과 잘 어울렸다.

## 4. 어려웠던 점 (Problem)

**VectorStore 구성 이유를 처음에 이해하지 못함**  
초반에 VectorStore를 왜 구성해야 하는지 명확히 이해하지 못해 다른 방식으로 접근하는 시행착오가 있었다. 임베딩이 텍스트를 수치 벡터로 변환하여 의미 기반 검색을 가능하게 한다는 개념을 직접 구현하면서 체득했다.

**JSON 및 GraphDB 데이터 활용 방법 미숙**  
`flavor_text.json`을 어떻게 전처리해서 인덱싱할지, Neo4j 그래프 데이터를 어떻게 설계하고 Cypher 쿼리로 탐색할지 처음에는 막막했다. 특히 `AGAINST` 관계로 듀얼 타입 복합 배율을 표현하는 방법을 찾는 데 시간이 걸렸다.

**RAG, VectorStore, LangGraph 역할 혼동**  
RAG(검색 증강 생성), VectorStore(벡터 저장소), LangGraph(Agent 흐름 제어)가 각각 어떤 역할을 하는지 처음에는 경계가 모호했다. 전체 시스템을 직접 만들어가면서 각 기술의 역할과 관계를 명확히 구분할 수 있게 되었다.

**스트리밍과 CRAG 충돌 처리**  
`astream_events`에서 `agent` 노드 토큰만 골라내고, rewriter 내부 호출 토큰을 필터링하는 작업이 예상보다 까다로웠다. 태그(`nosync`, `rewriter`)와 `langgraph_node` 메타데이터를 함께 활용하여 해결했다.

## 5. 개선할 점 (Try)

**GraphDB 설계를 사전에 더 구체적으로**  
Neo4j 그래프 스키마(노드·관계·속성)를 개발 전에 충분히 설계했더라면 Cypher 쿼리 작성과 데이터 활용이 훨씬 수월했을 것이다. 다음에는 ERD를 작성하듯 그래프 스키마를 먼저 도식화한 뒤 개발에 착수하고 싶다.

**데이터 구성 전략 수립**  
어떤 데이터를 정형 DB에, 어떤 데이터를 VectorStore에, 어떤 데이터를 GraphDB에 저장할지 기준을 초반에 명확히 정했다면 중복 작업과 시행착오를 줄일 수 있었을 것이다.

**BM25 인덱스 비동기 초기화**  
현재 BM25 인덱스는 첫 검색 요청 시 동기적으로 빌드되어 초기 응답이 느릴 수 있다. 서버 시작 시 백그라운드에서 미리 빌드하는 방식으로 개선하고 싶다.

## 6. 배운 것

**전체 그림이 보이면 문제가 보인다**  
프로젝트가 완성된 모습을 보고 나서야 초반에 무엇이 문제였는지 명확히 알 수 있었다. 구현 중에는 보이지 않던 설계 이슈나 비효율이 완성 후에 드러났다.

**Git 사용 능력 향상**  
협업 과정에서 브랜치 관리, 충돌 해결, 커밋 단위 구성 등 실전 Git 사용법을 익혔다.

**기술 스택 역할 구분**  
RAG, VectorStore, GraphDB, LangGraph, Embedding 등 각 기술이 전체 시스템에서 어떤 역할을 담당하는지 명확히 구분할 수 있게 되었다. 단순히 사용법을 아는 것을 넘어 왜 이 기술을 선택했는지 설명할 수 있는 수준이 되었다.

**graceful degradation 설계**  
BM25 빌드 실패, Neo4j 연결 실패 등 개별 컴포넌트가 실패해도 나머지가 정상 동작하도록 각 채널을 독립적으로 처리하는 설계 패턴을 익혔다. 전체 서비스가 단일 장애로 멈추지 않도록 방어적으로 코드를 작성하는 습관이 생겼다.
