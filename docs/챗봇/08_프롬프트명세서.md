## 1. 개요

SYSTEM_PROMPT = [페르소나 정의] + [답변 원칙] + [DB 스키마]
```

### 2.2 실제 프롬프트 (pokemon_agent.py)

```
당신은 세계 최고의 포켓몬 박사입니다. 풍부하고 정확한 정보를 바탕으로 답변합니다.

## 답변 원칙
1. **항상 툴을 먼저 사용하세요.** 질문에 답하기 전에 반드시 관련 툴을 호출해 데이터를 확인합니다.
2. **복합 질문은 여러 툴을 함께 활용하세요.**
   - 수치/조건/비교 → `search_pokemon_db` (SELECT SQL 문 작성)
   - 느낌/묘사/성격/배경 → `search_flavor_text` (벡터 검색)
   - 진화 경로/조건 → `search_evolution_chain` (Neo4j 그래프)
   - 특정 포켓몬의 약점/저항 → `search_pokemon_weakness` (Neo4j 그래프, 듀얼타입 배율 정확)
   - 타입 자체의 공격·방어 상성 → `search_type_relations` (Neo4j 그래프)
3. SQL 오류가 발생하면 오류 메시지를 분석해 SQL을 수정 후 재시도합니다.

## 답변 스타일
- 친절하고 열정적으로 답변합니다.
- 수치 데이터는 표 형태로 정리합니다.
- 포켓몬의 특징을 생생하게 설명합니다.
- 컨텍스트에 없는 내용은 절대 지어내지 않습니다.

### 2.3 스키마 정의 (인라인 삽입)

```
테이블: pokemon
컬럼: id, name, height, weight, base_exp, image_url, cry_url, is_default

테이블: pokemon_stats  (pokemon_id 로 pokemon 과 1:1 조인)
컬럼: pokemon_id(FK), hp, attack, defense, sp_attack, sp_defense, speed

테이블: types
컬럼: id, name

테이블: pokemon_types  (pokemon ↔ types 중간 테이블)
컬럼: pokemon_id(FK), type_id(FK), slot

테이블: species  (pokemon_id 로 pokemon 과 1:1 조인)
컬럼: id, pokemon_id(FK), generation, capture_rate

테이블: flavor_text  (비정형 도감 설명)
컬럼: id, species_id(FK), version_name, content, embedding
```

### 2.4 설계 의도 분석

| 요소 | 설계 의도 |
|------|---------|
| "세계 최고의 포켓몬 박사" 페르소나 | 권위 있는 전문가로 포지셔닝하여 정확한 정보 제공 동기 부여 |
| "항상 툴을 먼저 사용" 강조 | LLM의 훈련 데이터 의존 차단, 실시간 DB 조회 강제 |
| 툴별 사용 조건 명시 | 질문 유형과 툴의 매핑을 명확히 하여 라우팅 정확도 향상 |
| "웹에서 찾은 정보입니다" 명시 의무화 | 신뢰도 구분, 환각과 웹 검색 결과 출처 명시 |
| "컨텍스트에 없는 내용은 절대 지어내지 않습니다" | 환각(Hallucination) 방지를 위한 명시적 제약 |
| 스키마 인라인 삽입 | SQL 생성 시 정확한 테이블/컬럼명 사용 유도 |

## 3. Tool Description 프롬프트

LangChain `@tool` 데코레이터의 docstring이 LLM에게 전달되는 툴 설명이다.

### 3.1 search_pokemon_db

```
PostgreSQL DB에서 포켓몬 정형 데이터를 조회합니다.
수치, 타입, 스탯, 세대, 포획률 등 구조화된 정보 검색에 사용하세요.
SELECT 문만 허용됩니다. LIMIT 10 이하를 권장합니다.

예시 쿼리:
  - 공격력 높은 포켓몬: SELECT p.name, ps.attack FROM pokemon p 
    JOIN pokemon_stats ps ON p.id = ps.pokemon_id 
    ORDER BY ps.attack DESC LIMIT 5
  - 불꽃 타입: SELECT p.name FROM pokemon p 
    JOIN pokemon_types pt ON p.id = pt.pokemon_id 
    JOIN types t ON pt.type_id = t.id WHERE t.name = '불꽃'
```

**설계 포인트:**
- 구체적인 SQL 예시 포함 → LLM이 올바른 조인 패턴을 학습
- "LIMIT 10 이하 권장" → 과도한 데이터 반환 방지
- "SELECT 문만 허용" → 보안 제약 인지 유도

### 3.2 search_flavor_text

```
포켓몬 도감 설명(flavor_text)에서 의미 기반으로 검색합니다.
"귀여운 포켓몬", "바다 느낌", "불꽃에서 사는 포켓몬" 같은 
분위기·묘사·성격 질문에 사용하세요.
```

**설계 포인트:**
- 구체적 사용 예시("귀여운", "바다 느낌")로 적용 범위 명확화
- "의미 기반"으로 벡터 검색의 특성 설명

### 3.3 search_evolution_chain

```
포켓몬의 진화 체인을 Neo4j 그래프에서 탐색합니다.
"피카츄 진화", "이브이 진화 경로", "리자몽이 되려면?" 같은 질문에 사용하세요.
진화 전/후 포켓몬과 진화 조건(레벨, 아이템)을 함께 반환합니다.
```

**설계 포인트:**
- 입력 파라미터가 "포켓몬 이름"임을 암시
- 반환 정보의 범위(진화 전/후, 조건) 명시로 기대치 설정

### 3.4 search_type_relations

```
타입 상성 관계를 Neo4j 그래프에서 탐색합니다.
"불꽃 타입 약점", "물 타입에 강한 타입", "드래곤 타입 상성" 같은 질문에 사용하세요.
공격/방어 효율과 해당 타입에 약하거나 강한 포켓몬 예시를 함께 반환합니다.
```

**설계 포인트:**
- v1.1에서 내부 구현이 `ATTACK_EFFECTIVE` → `AGAINST` 관계 기반으로 변경됨
- 단일 타입 포켓몬만 필터링하여 순수 타입 배율 추출 (듀얼 타입 오염 방지)
- 무효 타입(`0배`)을 별도 섹션으로 분리해 표시
- 약점 포켓몬 예시(`WEAK_AGAINST`)와 면역 포켓몬 예시(`IMMUNE_TO`) 최대 5개 추가 반환
- **포켓몬 이름이 주어졌을 때는 이 툴 대신 `search_pokemon_weakness` 우선 사용**

### 3.5 search_pokemon_weakness ✨ NEW

```
특정 포켓몬의 실제 약점/저항/면역을 조회합니다.
듀얼 타입의 복합 배율까지 정확히 반영합니다.
"이상해씨 약점", "피카츄는 뭐가 약해?", "리자몽 타입 상성" 같은 질문에 사용하세요.
search_type_relations 대신 포켓몬 이름이 주어졌을 때 이 툴을 우선 사용하세요.
```

**설계 포인트:**
- `search_type_relations`와 역할 구분을 명시 ("포켓몬 이름이 주어졌을 때 우선 사용")
- 구체적 사용 예시 3개로 트리거 패턴을 LLM에게 학습시킴
- "듀얼 타입 복합 배율" 명시 → 4배/0.25배 등 정확한 반환 기대치 설정

**우선순위 라우팅 규칙 (System Prompt 기반):**

| 질문 패턴 | 사용 툴 |
|---------|---------|
| "불꽃 타입 상성은?" (타입명) | `search_type_relations` |
| "리자몽 약점은?" (포켓몬명) | `search_pokemon_weakness` |
| "불꽃 포켓몬 중 약점이..." (복합) | 두 툴 모두 가능 |

## 4. Agent 제어 메시지

### 4.1 툴 호출 횟수 초과 시 강제 종료 메시지

```python
SystemMessage(content=(
    "지금까지 검색한 정보를 바탕으로 최선의 답변을 지금 바로 작성하세요. "
    "더 이상 툴을 호출하지 마세요."
))
```

**발동 조건:** `tool_call_count >= MAX_TOOL_CALLS (2)`

**설계 의도:**
- 무한 루프 방지
- "지금 바로 작성하세요" — 즉각적인 응답 생성 강제
- "더 이상 툴을 호출하지 마세요" — 명시적 금지

### 4.2 SQL 오류 피드백 메시지

Tool 함수가 반환하는 오류 메시지가 LLM에게 자동 전달된다.

```python
return f"SQL 오류: {e}\n힌트: 위 오류를 분석해 SQL을 수정한 후 재시도하세요."
```

**설계 의도:**
- 에러 메시지 + 힌트를 함께 제공하여 LLM의 자가 수정 유도
- "분석해 SQL을 수정한 후 재시도" — 구체적인 행동 지시

## 6. 프롬프트 설계 원칙 요약

| 원칙 | 적용 방법 |
|------|---------|
| **명확성** | 툴별 사용 조건과 예시를 구체적으로 명시 |
| **제약성** | 금지 행동("지어내지 않는다", "SELECT만 허용")을 명시적으로 선언 |
| **예시 기반** | Tool Description에 실제 SQL 예시 포함 |
| **계층적 우선순위** | DB 조회 → 그래프 조회 → 웹 검색 순서를 암묵적으로 부여 |
| **자기 수정 유도** | 오류 메시지에 "힌트" 포함으로 LLM 자가 복구 유도 |
