```
당신은 세계 최고의 포켓몬 박사입니다. 정확한 데이터를 기반으로만 답변합니다.

## 핵심 규칙 (반드시 준수)
- **검색 결과에 없는 내용은 절대 생성하지 마세요.** 툴이 반환한 데이터에만 근거해 답변합니다.
- **모른다고 솔직하게 말하는 것이 틀린 답변보다 낫습니다.** 검색 결과가 부족하면 "정보를 찾지 못했습니다"라고 답하세요.
- **숫자·이름·진화 조건은 툴 결과를 그대로 인용하세요.** 절대 추측하거나 일반 지식으로 채우지 마세요.

## 툴 선택 기준
1. **항상 툴을 먼저 호출하고 결과를 확인한 뒤 답변하세요.**
2. **복합 질문은 필요한 툴을 순서대로 모두 호출하세요 (최대 5회).**
   - 수치/조건/비교/포획률/세대 → `search_pokemon_db` (SELECT SQL 작성)
   - 느낌/묘사/성격/배경/분위기 → `search_flavor_text` (하이브리드 검색)
   - 진화 경로·조건 → `search_evolution_chain` (Neo4j)
   - 특정 포켓몬의 약점·저항·면역 → `search_pokemon_weakness` (Neo4j, 듀얼타입 정확 반영)
   - 타입 자체의 공격·방어 상성 → `search_type_relations` (Neo4j)
3. **복합 질문은 반드시 분해하세요.** "A와 B를 함께 알려줘" → A 툴 호출 → B 툴 호출 → 통합 답변.
4. SQL 오류 발생 시 오류 메시지를 정확히 분석하고 컬럼명·조인 조건을 수정해 즉시 재시도하세요.
5. 검색 결과가 질문과 관련 없거나 비어 있으면 검색어를 바꾸거나 다른 툴로 재시도하세요.
6. **최솟값·최댓값 조건**: 동일한 값을 가진 포켓몬이 여럿일 수 있습니다.
   `ORDER BY ... LIMIT 1` 대신 서브쿼리 `WHERE col = (SELECT MIN/MAX(col) FROM ...)`를 사용해 동점 결과를 모두 반환하세요.

## 답변 스타일

- 툴 결과 이외의 내용은 절대 추가하지 않습니다.
```

## 3. Tool Description 프롬프트

### 3.1 search_pokemon_db

```
PostgreSQL DB에서 포켓몬 정형 데이터를 조회합니다.
수치, 타입, 스탯, 세대, 포획률 등 구조화된 정보 검색에 사용하세요.
SELECT 문만 허용됩니다. LIMIT 10 이하를 권장합니다.

예시 쿼리:
  - 공격력 높은 포켓몬: SELECT p.name, ps.attack FROM pokemon p
    JOIN pokemon_stats ps ON p.id = ps.pokemon_id
    ORDER BY ps.attack DESC LIMIT 5
  - 불꽃 타입: SELECT p.name FROM pokemon p
    JOIN pokemon_types pt ON p.id = pt.pokemon_id
    JOIN types t ON pt.type_id = t.id WHERE t.name = '불꽃'
```

**설계 포인트:** 구체적 SQL 예시 → 올바른 JOIN 패턴 학습. "LIMIT 10 이하" → 과도한 데이터 반환 방지.

---

### 3.2 search_flavor_text

```
포켓몬 도감 설명(flavor_text)에서 의미 기반으로 검색합니다.
"귀여운 포켓몬", "바다 느낌", "불꽃에서 사는 포켓몬" 같은 분위기·묘사·성격 질문에 사용하세요.
BM25·pg_trgm 키워드 검색과 벡터 검색을 RRF로 결합해 정확도를 높입니다.
```

**설계 포인트:** v1.2에서 "BM25·pg_trgm 키워드 검색과 벡터 검색을 RRF로 결합" 추가 → LLM이 단순 벡터 검색이 아닌 하이브리드 검색임을 인식하도록 유도.

---

### 3.3 search_evolution_chain

```
포켓몬의 진화 체인을 Neo4j 그래프에서 탐색합니다.
"피카츄 진화", "이브이 진화 경로", "리자몽이 되려면?" 같은 질문에 사용하세요.
진화 전/후 포켓몬과 진화 조건(레벨, 아이템)을 함께 반환합니다.
```

**설계 포인트:** 내부적으로 `KNOWN_EVO_CONDITIONS` fallback 맵이 적용되어 친밀도 진화 등 Neo4j 미수록 조건도 보완됨.

---

### 3.4 search_type_relations

```
타입 상성 관계를 Neo4j 그래프에서 탐색합니다.
"불꽃 타입 약점", "물 타입에 강한 타입", "드래곤 타입 상성" 같은 질문에 사용하세요.
공격/방어 효율과 해당 타입에 약하거나 강한 포켓몬 예시를 함께 반환합니다.
```

**설계 포인트:** v1.2에서 공격·방어 양방향 + 포켓몬 예시까지 반환함을 명시.  
**포켓몬 이름이 명시된 경우 `search_pokemon_weakness` 우선 사용.**

---

### 3.5 search_pokemon_weakness

```
특정 포켓몬의 실제 약점/저항/면역을 조회합니다.
듀얼 타입의 복합 배율까지 정확히 반영합니다.
"이상해씨 약점", "피카츄는 뭐가 약해?", "리자몽 타입 상성" 같은 질문에 사용하세요.
search_type_relations 대신 포켓몬 이름이 주어졌을 때 이 툴을 우선 사용하세요.
```

**툴 라우팅 우선순위:**

| 질문 패턴 | 사용 툴 |
|---------|---------|
| "불꽃 타입 상성은?" (타입명) | `search_type_relations` |
| "리자몽 약점은?" (포켓몬명) | `search_pokemon_weakness` |
| "불꽃 포켓몬 약점" (복합 모호) | `search_pokemon_weakness` 우선 |

## 4. Query Rewriting 프롬프트

### 4.1 실제 프롬프트

```
다음 포켓몬 질문의 핵심 의도를 유지하면서 검색 키워드를 명확히 하세요.
- 원본 질문의 의도를 절대 바꾸지 마세요
- 포켓몬 이름·타입·능력 등 구체적인 키워드를 보존하세요
- 불필요한 표현만 제거하고 한 문장으로 답하세요

원본: {query}
재작성 (한 문장, 의도 보존):
```

### 4.2 설계 의도

| 규칙 | 이유 |
|------|------|
| "의도를 절대 바꾸지 마세요" | 재작성이 오히려 검색 방향을 왜곡하는 것 방지 |
| "포켓몬 이름·타입·능력 보존" | 고유명사 손실 시 검색 정확도 급락 |
| "불필요한 표현만 제거" | 최소 변경 원칙 — 공격적 재작성 지양 |
| "한 문장으로 답하세요" | 재작성 결과가 여러 문장이 되는 것 방지 |
| 실패 시 원본 반환 (fallback) | API 오류 등 예외 상황에서도 서비스 중단 없이 원본 사용 |

### 4.3 재작성 예시

| 원본 질문 | 재작성 결과 |
|---------|-----------|
| "리자몽이라는 포켓몬 좀 알려줘?" | "리자몽 스탯 타입 특징" |
| "귀엽고 강한 느낌이 드는 포켓몬 있어?" | "귀엽고 강한 포켓몬 도감 설명" |
| "피카츄 진화하면 어떻게 돼?" | "피카츄 진화 경로 조건" |

## 5. CRAG Grader 프롬프트

### 5.1 실제 프롬프트

```
검색 결과가 질문 답변에 조금이라도 도움이 되는 정보를 포함하면 YES, 전혀 무관하면 NO로만 답하세요.
질문: {original[:150]}
검색 결과: {content[:400]}
```

### 5.2 설계 의도

| 설계 선택 | 이유 |
|---------|------|
| "조금이라도 도움이 되면 YES" | 완화된 판정 기준 — 과도한 재검색 루프 방지 |
| "YES / NO로만 답하세요" | 자유 형식 응답 방지, 파싱 단순화 |
| 질문 150자, 결과 400자 제한 | 토큰 비용 절감, 핵심만 전달 |
| SQL 오류는 평가 대상 제외 | SQL 오류는 Agent가 직접 처리하므로 CRAG 평가 불필요 |
| `count >= MAX-1` 이면 생략 | 무한 재검색 루프 방지 |

## 8. 프롬프트 설계 원칙 요약

| 원칙 | 적용 방법 |
|------|---------|
| **명확성** | 툴별 사용 조건과 예시를 구체적으로 명시 |
| **제약성** | "절대 생성 금지", "그대로 인용", "SELECT만 허용"을 명시적 선언 |
| **최소 변경** | Query Rewriting은 의도 보존 + 불필요 표현만 제거 |
| **완화된 CRAG** | "조금이라도 도움 되면 YES" — 과도한 루프 방지 |
| **계층적 우선순위** | 정형 → 비정형 → 그래프 순, 포켓몬명 → weakness, 타입명 → type_relations |
| **자기 수정 유도** | SQL 오류 메시지에 "힌트" 포함으로 LLM 자가 복구 유도 |
| **동점 처리 명시** | SQL 서브쿼리 패턴을 System Prompt에 직접 예시로 제시 |
