# 프로젝트 회고록 (Retrospective)

**프로젝트명:** 포켓몬 AI 챗봇 (Pokémon AI Chatbot)  
**작성일:** 2025-05-14  
**작성자:** -  
**담당 기능:** 포켓몬 챗봇 백엔드 (Agent, VectorStore, GraphDB, ChatHistory)

---

## 1. 프로젝트 개요

사용자가 포켓몬에 대해 자연어로 질문하면 AI가 답해주는 챗봇을 만들었다.  
단순히 DB에서 정보를 꺼내주는 게 아니라, VectorStore와 GraphDB를 함께 활용해서 도감 설명부터 진화 경로, 타입 상성까지 다양한 질문에 대답할 수 있도록 구성했다.  
5명이서 팀을 이뤘고, 나는 챗봇 백엔드 전체를 담당했다.

---

## 2. 담당 역할 및 작업

### 2.1 담당 모듈

| 파일 | 역할 |
|------|------|
| `ingest.py` | `flavor_text.json`을 PGVector VectorStore에 임베딩하여 인덱싱 |
| `chat_history.py` | 채팅 세션과 메시지를 PostgreSQL에 저장·불러오기 |
| `pokemon_agent.py` | LangGraph Agent 설계, Hybrid Search 구현, Tool 작성 |
| `pokemon_neo4j.py` | Neo4j 연동, 진화 체인·타입 상성·약점 Tool 구현 |

### 2.2 주요 작업 상세

**`pokemon_agent.py` — LangGraph Agent 및 Hybrid Search**
- LangGraph 3-노드 구조 설계: `agent → tools → crag → agent` 루프
- `agent_node`: 질문 재작성(Query Rewriting) 후 LLM이 어떤 툴을 쓸지 판단
- `crag_check_node`: 검색 결과가 엉뚱하면 다시 검색하도록 유도
- PGVector + BM25 + pg_trgm 세 가지 검색 결과를 RRF로 합쳐 Hybrid Search 구현
- `search_pokemon_db`, `search_flavor_text` Tool 작성

**`pokemon_neo4j.py` — GraphDB 연동**
- 진화 체인 탐색(`search_evolution_chain`): 진화 전후와 조건까지 반환
- 타입 상성(`search_type_relations`): 공격·방어 양방향 조회
- 개체별 약점/저항/면역(`search_pokemon_weakness`): 듀얼 타입 복합 배율까지 반영
- DB에 없는 친밀도 진화 조건은 `KNOWN_EVO_CONDITIONS`로 직접 보완

**`ingest.py` — 임베딩 파이프라인**
- `flavor_text.json` 읽어서 Document로 변환 후 PGVector에 저장
- 이미 데이터가 있으면 다시 넣지 않도록 중복 방지 처리

**`chat_history.py` — 채팅 이력 관리**
- `chat_sessions` / `chat_messages` 테이블 설계 및 CRUD 구현
- 어떤 툴을 썼는지(`used_tools`)도 같이 저장해서 디버깅에 활용

## 3. 잘 된 점 (Keep)

**VectorDB + GraphDB를 목적에 맞게 나눠서 쓴 것**  
처음에는 그냥 DB 하나로 다 하면 되지 않나 싶었는데, 도감 설명 같은 텍스트는 벡터 검색으로, 진화나 타입 상성 같은 관계 데이터는 그래프로 따로 처리하니까 훨씬 정확한 답변이 나왔다. 각각의 강점을 살려서 쓴 게 이번 프로젝트에서 가장 잘한 부분이라고 생각한다.

**RAG 구조를 단계별로 쌓아올린 것**  
처음에는 그냥 벡터 검색만 했는데, BM25랑 pg_trgm을 추가하고 RRF로 합치고, 거기에 CRAG까지 붙이면서 검색 품질이 눈에 띄게 좋아졌다. 한 번에 완성한 게 아니라 문제가 생길 때마다 하나씩 붙여나간 결과라서 더 뿌듯했다.

**LangGraph 노드 역할을 깔끔하게 나눈 것**  
`agent → tools → crag` 흐름으로 각 노드가 뭘 하는지 명확하게 분리했다. 덕분에 오류가 생겨도 어느 노드에서 문제가 생긴 건지 바로 파악할 수 있어서 디버깅이 편했다.

---

## 4. 어려웠던 점 (Problem)

**VectorStore가 왜 필요한지 처음에 몰랐던 것**  
솔직히 초반에는 그냥 SQL로 검색하면 되지 않나 싶어서 SQL 방식으로 먼저 짰다가 다 갈아엎었다. 임베딩이 뭔지, 왜 벡터로 저장해야 하는지를 먼저 이해하고 시작했어야 했는데, 일단 코드부터 쓰려다 보니 시간을 많이 날렸다.

**데이터를 어디에 어떻게 넣어야 하는지 감이 없었던 것**  
JSON 데이터를 어떻게 전처리해서 VectorStore에 넣는지, Neo4j에는 어떤 관계를 어떻게 설계해야 하는지 처음에는 진짜 막막했다. 특히 듀얼 타입 포켓몬의 복합 배율을 Neo4j에서 어떻게 표현해야 하는지 찾는 데 꽤 오래 걸렸다.

**RAG, VectorStore, LangGraph가 각각 뭐하는 건지 헷갈렸던 것**  
처음에는 셋이 다 비슷한 거 아닌가 싶었다. 직접 만들어보면서 RAG는 검색해서 LLM에 넘겨주는 방식, VectorStore는 그 검색을 위한 저장소, LangGraph는 전체 흐름을 제어하는 프레임워크라는 걸 겨우 구분하게 됐다.

## 5. 개선할 점 (Try)

**GraphDB 설계를 먼저 그리고 시작하기**  
Neo4j 노드랑 관계를 머릿속으로만 생각하고 바로 코딩했는데, 중간에 구조를 여러 번 바꾸면서 시간이 많이 걸렸다. 다음에는 ERD 그리듯이 그래프 스키마를 먼저 종이에라도 그려놓고 시작할 것이다.

**어떤 데이터를 어느 DB에 넣을지 먼저 정하기**  
정형 데이터는 PostgreSQL, 텍스트는 VectorStore, 관계는 Neo4j라는 기준을 프로젝트 초반에 팀이랑 같이 정했더라면 삽질을 많이 줄일 수 있었을 것 같다. 다음에는 설계 단계에서 이 부분을 꼭 먼저 정하고 넘어가겠다.

**코드를 직접 써보는 연습 더 하기**  
이번에 코드를 읽는 건 어느 정도 됐는데 직접 처음부터 짜는 건 많이 힘들었다. 앞으로는 예제를 보더라도 그냥 따라 치는 게 아니라 이해하고 나서 빈 파일에 처음부터 써보는 연습을 해야겠다.

## 6. 배운 것

**각 기술이 왜 필요한지 이제는 설명할 수 있다**  
LangGraph, RAG, VectorStore, GraphDB, 임베딩… 프로젝트 전에는 이름만 들어봤던 것들이 이제는 각각 어떤 역할을 하고 왜 쓰는지 말할 수 있게 됐다. 그냥 쓸 줄 아는 것과 설명할 수 있는 것은 다르다는 걸 이번에 느꼈다.

**전체 구조를 먼저 보고 시작해야 한다는 것**  
완성하고 나서 보니까 처음에 전체 흐름을 제대로 이해하고 시작했으면 훨씬 빨리 끝났을 것 같았다. 앞으로는 코딩 전에 전체 아키텍처를 그려보는 시간을 꼭 갖겠다.

**Git이 이렇게 중요한 줄 몰랐다**  
혼자 할 때는 그냥 저장하면 됐는데, 팀 프로젝트에서는 브랜치 관리나 충돌 해결이 진짜 중요하다는 걸 몸소 체험했다. 처음에는 머지 충돌 날 때마다 당황했는데 이제는 어느 정도 익숙해졌다.

## 7. 마무리 소감

솔직히 많이 반성되는 한 주였다. 코드를 눈으로 읽는 건 되는데 막상 내가 직접 짜려고 하면 손이 안 움직이는 순간이 너무 많았다. 알고 있다고 생각했는데 실제로는 모르고 있었던 것들이 많았던 것 같다. 그래도 이번 프로젝트 덕분에 LangGraph랑 RAG 구조가 어떻게 맞물려 돌아가는지는 제대로 이해하게 됐고, 다음에는 개념을 먼저 잡고 설계한 뒤에 코딩하는 순서를 꼭 지켜야겠다고 생각했다. 아쉬운 게 많이 남는 프로젝트였지만, 그만큼 배운 것도 많았다.
