## 1. 프로젝트 소개

### 한 줄 소개

> **포켓몬 세계관을 LLM·RAG·그래프 DB로 구현한 풀스택 AI 플랫폼**

### 기획 배경

포켓몬은 1,025종의 포켓몬, 18종 타입, 수백 개의 특성과 기술로 이루어진 **구조적으로 복잡한 도메인**입니다.

단순한 키워드 검색으로는 "내 팀의 약점을 보완할 포켓몬이 뭐야?" 같은 질문에 답할 수 없습니다.  
이 문제를 **그래프 DB + 벡터 DB + LLM**으로 해결하는 것이 이 프로젝트의 출발점입니다.

### 해결하려 한 문제

| 문제 | 우리의 접근 |
|---|---|
| 타입 상성처럼 관계 기반 추론이 필요한 질문 | Neo4j 그래프 DB로 관계 직접 계산 |
| "이 팀에 어울리는 포켓몬" 같은 시맨틱 질문 | pgvector MMR 검색으로 의미 기반 검색 |
| LLM이 수치를 틀리는 환각 문제 | 구조화 DB 결과를 프롬프트에 주입, LLM 추론 금지 |
| 챗봇이 모르는 최신 정보 | Tavily 웹 검색 폴백 |

## 3. 기술 스택

```
Frontend     Streamlit · streamlit-cookies-controller
Backend      FastAPI · SQLAlchemy · Pydantic
AI / LLM     LangChain · LangGraph · LangSmith
LLM 모델     OpenAI GPT-4o-mini · Groq llama-3.3-70b
임베딩/검색  pgvector (MMR) · BM25 · Tavily
관계형 DB    PostgreSQL 16 + pgvector
그래프 DB    Neo4j 5 + APOC + GDS
인증         GitHub OAuth 2.0
인프라       Docker · Docker Compose
```

### 왜 이 스택인가?

- **Neo4j**: 포켓몬 타입 상성은 18×18 행렬 관계 — 관계형 DB보다 그래프가 자연스럽고 Cypher 쿼리가 직관적
- **pgvector**: PostgreSQL 확장으로 기존 ORM과 동일 DB에서 벡터 검색 가능 → 인프라 단순화
- **LangGraph**: 단순 체인이 아닌 상태 기반 9-노드 워크플로우 — 각 단계 교체·디버깅 용이
- **Groq**: 배틀 봇 실시간 전략 판단 — OpenAI 대비 추론 속도 우선

## 6. 핵심 기술 — LangGraph Hybrid RAG

### 왜 Hybrid RAG인가?

"팀의 약점을 보완할 포켓몬" 이라는 질문에는 두 가지 정보가 모두 필요합니다.

| 정보 유형 | 수단 | 예시 |
|---|---|---|
| **구조적 관계** — 타입 상성, 약점, 종족값 | Neo4j Cypher | 불꽃팀의 약점은 바위·물·땅 → 이를 막는 후보 계산 |
| **시맨틱 설명** — 포켓몬 특징, 역할, 배치 이유 | pgvector MMR | "방어형 포켓몬" 관련 도감 설명 근거 |

### Graph-guided 순차형 (병렬형 아님)

```
5마리 입력
    │
    ▼
Neo4j Cypher ← 약점 / 저항 / 기술 커버리지 계산
    │ 계산 결과로 검색 쿼리 생성
    ▼
pgvector MMR (k=20, fetch_k=50)
    │ 시맨틱 근거 문서
    ▼
Hybrid Score 계산 → LLM 한국어 해설 생성
```

Graph DB가 먼저 계산하고, 그 결과로 Vector 검색어를 만드는 **순차형** 구조입니다.  
Vector가 Graph를 보완하지, 두 경로가 대등하게 병렬로 흐르지 않습니다.

### 9-노드 LangGraph 워크플로우

| 노드 | 역할 |
|---|---|
| Node 1 · supervisor | 5마리 존재 검증 · 요청 유형 분류 (분석/추천) |
| Node 2 · select_graph_tool | Neo4j 쿼리 도구 선택 |
| Node 3 · execution_graph_tool | Cypher 실행 — 약점·저항·커버리지 계산 |
| Node 4 · vector_search | Graph 결과 → 검색 쿼리 → pgvector MMR |
| Node 5 · evaluate_with_llm | Graph + Vector 컨텍스트 결합 |
| Node 6 · hybrid_scorer | 목적별 가중치로 점수 계산 |
| Node 7 · answer_generator | LLM 한국어 해설 생성 |
| Node 8 · save_analysis | team_build_logs JSONB 저장 |
| Node 9 · save_recommendation | 추천 결과 JSONB 저장 |

### Hybrid Score — 목적별 가중치

단순히 고정 비율을 쓰지 않고 **단계마다 다른 가중치**를 적용했습니다.

| 단계 | Graph | Vector | 설계 근거 |
|---|---|---|---|
| 덱 분석 | **70%** | 30% | 타입 상성 계산이 핵심, Vector는 설명 보강 |
| 포켓몬 추천 | **80%** | 20% | 약점 보완·종족값·커버리지 계산이 최우선 |
| AI 해설 생성 | **60%** | **40%** | 계산 근거 우선, Vector로 설명 풍부함 확보 |

**graph_score 산출**: 약점 보완(최대 125pt) + 종족값(5pt) + 기술 커버리지(20pt) = 최대 150pt → 0~100 정규화  
타입 중복 후보는 최대 -40pt 감점

## 7. 핵심 기술 — 환각 방지 전략

LLM이 포켓몬 타입 배율이나 스탯 수치를 틀리는 것은 치명적입니다.  
**파이프라인 수준 + 프롬프트 수준** 이중 제어로 막았습니다.

| 전략 | 적용 위치 | 효과 |
|---|---|---|
| **근거 기반 생성 강제** | 팀 빌더·챗봇 프롬프트 | DB 검색 결과만 사용, 컨텍스트 외 생성 금지 지시 |
| **MMR 다양성 검색** | pgvector 검색 | 중복 컨텍스트 제거, 단일 정보 과의존 방지 |
| **Hybrid Reranking** | 팀 빌더 추천 | 목적별 가중치로 그래프 관계 우선 반영 |
| **SQL 자동 재시도** | 챗봇 SQL 도구 | 실패 시 3회 수정·재실행 → 웹 검색 폴백 |
| **Cross-encoder Re-ranking** | 챗봇 다중 도구 결과 | 도구 간 결과 통합 후 관련성 재평가 |
| **원문 컨텍스트 저장** | 팀 빌더 JSONB | 답변 + 근거를 함께 저장 → 추적 가능성 확보 |
| **Neo4j 구조화 데이터 우선** | 타입 상성 계산 | LLM 추론 없이 Cypher 결과 직접 사용 |

**핵심 프롬프트 지시:**
```
아래 [검색 컨텍스트]에 제공된 정보만을 근거로 답변하세요.
컨텍스트에 없는 정보는 "알 수 없습니다"로 답변하고 임의로 생성하지 마세요.
수치 데이터(스탯, 타입 배율)는 반드시 컨텍스트의 값을 그대로 사용하세요.
```

## 8. RAG 품질 평가 결과

### 측정 방법

- **Faithfulness** (환각률): 수동 샘플링 30건 — 근거 외 정보 포함 여부
- **Answer Relevancy**: 수동 샘플링 50건 — 관련성 3점 척도
- **Context Recall**: 정답 컨텍스트 포함률
- **Hybrid Score 유효성**: Graph-only vs Hybrid 팀 커버리지 비교

### 결과

| 지표 | 목표 | 측정치 |
|---|---|---|
| Faithfulness (환각 없음) | ≥ 90% | **90%** |
| Answer Relevancy | ≥ 85% | **92%** |
| Context Recall | ≥ 80% | **88%** |
| Hybrid vs Graph-only 커버리지 | 개선 | **+18%** 향상 |
| Retrieval 정확도 | ≥ 90% | **92%** |
| 첫 토큰 수신 지연 | ≤ 2초 | **평균 1.2초** |
| SQL 재시도 성공률 | ≥ 95% | **97%** |

**Hybrid RAG가 Graph-only 대비 팀 타입 커버리지 18% 향상**이 가장 의미 있는 수치입니다.  
Vector DB의 시맨틱 근거가 그래프 계산만으로는 잡지 못하는 포켓몬을 추천에 반영했습니다.

## 9. 어려웠던 점 & 배운 점

### 어려웠던 점

**Neo4j와 pgvector를 어떻게 결합할 것인가**

처음에는 두 DB를 병렬로 조회해서 점수를 합산하는 방식을 고려했습니다.  
그런데 Vector 검색의 쿼리 문장 자체가 Graph 결과(약점 타입, 추천 후보 조건)에서 나와야 의미가 있다는 것을 깨달았고,  
**Graph-guided 순차형**으로 설계를 바꿨습니다. 이게 훨씬 자연스럽고 결과도 좋았습니다.

**환각 방지와 답변 품질의 트레이드오프**

컨텍스트 외 생성을 강하게 금지하면 "모릅니다" 답변이 너무 많아지고,  
너무 느슨하면 수치 오류가 생겼습니다.  
LangSmith로 실제 환각 케이스를 trace로 역추적하면서 프롬프트를 반복 수정했습니다.

**Streamlit 세션의 스레드 안전 문제**

`ThreadPoolExecutor`로 팀 빌더 분석·추천을 병렬 호출할 때,  
스레드 내부에서 `st.session_state`를 읽으면 주 스레드와 격리되어 값이 None이 됩니다.  
`user_id`를 메인 스레드에서 미리 캡처 후 함수에 인수로 주입하는 방식으로 해결했습니다.

### 배운 점

- **LangGraph로 AI 파이프라인을 상태 기반으로 설계**하면 디버깅과 노드 단위 개선이 훨씬 쉽다
- **DB를 목적에 맞게 분리**하는 것이 단일 DB + 복잡한 쿼리보다 낫다
- **환각은 프롬프트 지시만으로 완전히 막기 어렵고**, 파이프라인 수준(구조화 DB 우선)과 병행해야 한다
- LangSmith 트레이싱을 초반부터 켜두면 **문제가 생겼을 때 노드 단위로 즉시 찾을 수 있다**

## 10. Q&A

> 발표 중 예상 질문

**Q. Graph DB와 Vector DB를 둘 다 쓰는 이유는?**  
A. 타입 상성처럼 명확한 관계 계산은 그래프가 정확하고 빠릅니다. 도감 설명 같은 자연어 유사도는 벡터가 필요합니다. 두 질문 유형이 모두 존재하므로 두 DB를 함께 씁니다.

**Q. 환각이 여전히 10% 발생하는 이유는?**  
A. 컨텍스트 자체에 정보가 없는 경우(미수집 데이터, 신규 포켓몬 등)에서 주로 발생합니다. DB 커버리지를 높이거나 "정보 없음" 응답을 명확히 출력하는 방향으로 개선할 수 있습니다.

**Q. LangGraph를 쓴 이유는 단순 LangChain 체인 대비 장점은?**  
A. 노드 단위 상태 관리로 중간 결과를 저장·전달할 수 있고, 특정 노드만 교체하거나 조건부 분기를 추가하기 쉽습니다. LangSmith에서 노드별 추적도 됩니다.

**Q. 피피고와 메인 앱의 연결은?**  
A. 현재는 독립적입니다. 피피고는 Groq API를 직접 호출하고, 메인 앱 백엔드와 연동하지 않습니다. 향후 메인 앱 API를 통해 포켓몬 DB 검색 기능을 피피고에 연결할 수 있습니다.
