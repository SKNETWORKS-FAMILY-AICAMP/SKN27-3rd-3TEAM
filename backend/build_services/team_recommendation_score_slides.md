# 팀 빌더 추천 점수 계산 설명

## Graph Score + Vector Score + Hybrid Score

이 문서는 팀 빌더가 6번째 포켓몬을 추천할 때  
점수가 어떤 기준으로 계산되는지 발표 자료처럼 이해하기 쉽게 정리한 문서입니다.

---

# 1. 추천 점수의 큰 흐름

팀 빌더 추천은 한 번에 점수를 바로 계산하지 않습니다.

1. 선택한 5마리의 팀 약점을 분석합니다.
2. 약점을 보완할 수 있는 후보 포켓몬을 Graph DB에서 찾습니다.
3. 후보별 Graph 점수를 계산합니다.
4. RAG 추천에서는 Vector 근거 점수를 추가합니다.
5. Graph 점수와 Vector 점수를 섞어 최종 Hybrid 점수를 만듭니다.

---

# 2. 추천 후보를 찾는 기준

후보는 아무 포켓몬이나 가져오지 않습니다.

현재 팀이 약한 공격 타입을 기준으로,  
그 타입을 저항하거나 무효화할 수 있는 포켓몬을 먼저 찾습니다.

```cypher
(candidate:Pokemon)-[:RESISTANT_TO|VERY_RESISTANT_TO|IMMUNE_TO]->(weakType:Type)
```

즉, 추천의 핵심 질문은 다음과 같습니다.

> "현재 팀이 약한 타입을 누가 대신 받아줄 수 있는가?"

---

# 3. Graph Score 공식

Graph Score는 Neo4j 관계와 포켓몬 속성으로 계산하는 기본 추천 점수입니다.

```text
graph_score =
    defensive_score
  + stat_score
  + coverage_score
  - duplicate_penalty
```

이 점수는 "현재 팀에 실제로 필요한 포켓몬인가?"를 판단하는 데 집중합니다.

---

# 4. Graph Score 세부 항목

| 항목 | 계산식 | 의미 |
| --- | --- | --- |
| defensive_score | min(보완 약점 타입 수, 5) * 25 | 팀 약점을 많이 막을수록 가산 |
| stat_score | min(base_total / 140, 5) | 기본 능력치를 보조 점수로만 가산 |
| coverage_score | min(기술 타입 수 * 1.5, 20) | 배울 수 있는 기술 타입이 다양할수록 가산 |
| duplicate_penalty | min(팀과 겹치는 타입 종류 수 * 20, 40) | 기존 팀 타입과 겹치면 강하게 감점 |

이 공식은 수학적으로 증명된 공식이 아니라,  
팀 빌딩 목적에 맞춰 만든 **규칙 기반 휴리스틱 점수**입니다.

---

# 4-1. 왜 휴리스틱 점수를 사용했는가

포켓몬 추천은 정답 라벨이 있는 지도학습 문제가 아닙니다.

따라서 현재 단계에서는 모델 학습보다,  
설명 가능하고 조정하기 쉬운 규칙 기반 점수가 적합합니다.

| 설계 기준 | 이유 |
| --- | --- |
| 설명 가능성 | 왜 추천했는지 사용자에게 바로 설명 가능 |
| 조정 용이성 | 팀 약점 보완, 능력치, 기술 폭의 비중을 쉽게 수정 가능 |
| Graph DB 활용 | 타입 상성 관계를 직접 점수화 가능 |
| 초기 프로젝트 적합성 | 학습 데이터 없이도 안정적으로 추천 가능 |

---

# 4-2. 점수 배율 설계 기준

Graph Score는 네 항목을 같은 비중으로 두지 않았습니다.

추천 목적이 "팀 약점 보완"이므로,  
약점 보완 점수가 가장 큰 영향력을 갖도록 설계했습니다.

| 항목 | 일반적인 점수 범위 | 설계 의도 |
| --- | ---: | --- |
| defensive_score | 0~125 | 추천의 핵심 기준 |
| stat_score | 0~5 | 기본 전투력 보조 보정 |
| coverage_score | 0~20 | 기술 폭 보조 반영 |
| duplicate_penalty | 0~-40 | 타입 중복 강한 감점 |

즉, 전체 비중은 대략  
**약점 보완 > 타입 중복 회피 > 기술 폭 > 능력치 보조** 순서입니다.

Graph Score의 설계상 최대값은 **150점**입니다.  
이는 `defensive_score 125점 + stat_score 5점 + coverage_score 20점`을 합친 값이며,  
`duplicate_penalty`는 감점 항목이므로 최대점 계산에서는 0점 감점으로 봅니다.

---

# 5. defensive_score

## 약점 보완 점수

현재 팀이 약한 타입을 후보 포켓몬이 얼마나 잘 받아줄 수 있는지 계산합니다.

```text
defensive_score = min(보완 가능한 약점 타입 개수, 5) * 25
```

예를 들어 후보가 `바위`, `얼음`, `전기`를 저항하거나 무효화하면:

```text
3개 * 25 = 75점
```

추천 로직에서 가장 중요한 점수입니다.

단, 보완 가능한 약점 타입이 6개 이상이어도 점수가 무한히 커지지 않도록  
최대 5개까지만 점수화합니다.

```text
defensive_score 최대값 = 5 * 25 = 125
```

---

# 5-1. 왜 25점을 곱하는가

25점은 약점 보완을 Graph Score의 핵심 비중으로 만들기 위한 값입니다.

추천 후보가 약점 타입 2개를 막으면 50점,  
3개를 막으면 75점을 받습니다.

```text
약점 1개 보완 = 25점
약점 2개 보완 = 50점
약점 3개 보완 = 75점
약점 4개 보완 = 100점
```

이렇게 하면 약점 2~3개를 보완하는 후보가  
단순히 능력치만 높은 후보보다 우선될 수 있습니다.

---

# 5-2. 25점의 비교 기준

25점은 다른 점수 항목과 비교했을 때 의도가 분명합니다.

| 비교 대상 | 의미 |
| --- | --- |
| 약점 1개 보완 25점 | 기술 커버리지 최대 20점보다 큼 |
| 약점 2개 보완 50점 | 타입 중복 최대 감점 40점보다 큼 |
| 약점 3개 보완 75점 | 능력치 최대 5점보다 훨씬 큼 |

즉, 25점은 총점의 절반을 항상 차지하게 하려는 값은 아닙니다.

대신 약점 2~3개를 보완할 때  
추천 순위를 실제로 뒤집을 수 있게 만든 조정값입니다.

핵심 의도는 **강한 포켓몬보다 필요한 포켓몬을 우선하기**입니다.

---

# 6. stat_score

## 기본 능력치 점수

포켓몬의 기본 능력치 합계인 `base_total`을 점수화합니다.

```text
stat_score = min(base_total / 140, 5)
```

예시:

```text
base_total = 600
600 / 140 = 4.29점
```

다만 너무 높은 종족값이 점수를 과도하게 지배하지 않도록 최대 5점으로 제한합니다.

---

# 6-1. 왜 base_total을 140으로 나누는가

포켓몬의 `base_total`은 보통 수백 단위입니다.  
그 값을 그대로 쓰면 능력치가 추천 점수를 전부 지배합니다.

그래서 `base_total / 140`으로 크게 줄여서  
능력치가 추천 순위를 뒤집는 핵심 기준이 아니라 보조 기준이 되도록 맞춥니다.

```text
base_total 420 -> 3.00점
base_total 560 -> 4.00점
base_total 700 -> 5.00점
```

이렇게 하면 능력치는 후보 간 체급 차이를 살짝 보정하지만,  
약점 보완이나 타입 다양성보다 앞서지 않습니다.

---

# 6-2. 왜 최대 5점인가

전설급 또는 초고성능 포켓몬이 능력치만으로 무조건 1등이 되는 것을 막기 위해서입니다.

이 추천 서비스는 "가장 강한 포켓몬"보다  
"현재 팀 약점을 가장 잘 메우는 포켓몬"을 추천해야 합니다.

| 후보 유형 | 능력치 점수 |
| --- | ---: |
| 일반 포켓몬 base_total 420 | 3.00점 |
| 강한 포켓몬 base_total 560 | 4.00점 |
| 매우 강한 포켓몬 base_total 700 이상 | 최대 5.00점 |

그래서 능력치는 중요하지만,  
약점 보완이나 타입 중복 회피보다 커지지 않게 제한했습니다.

---

# 7. coverage_score

## 기술 타입 커버리지 점수

후보 포켓몬이 배울 수 있는 기술 타입이 다양할수록 점수를 받습니다.

```text
coverage_score = min(기술 타입 수 * 1.5, 20)
```

예시:

```text
기술 타입 수 = 12개
12 * 1.5 = 18점
```

기술 타입이 많으면 상대 상황에 맞춰 대응할 수 있는 폭이 넓어집니다.

---

# 7-1. 왜 1.5를 곱하는가

기술 타입 커버리지는 중요하지만,  
약점 보완이나 기본 능력치보다 우선되면 안 됩니다.

그래서 기술 타입 1개당 1.5점만 부여합니다.

```text
기술 타입 4개  -> 6점
기술 타입 10개 -> 15점
기술 타입 14개 -> 20점 제한
```

1.5를 곱하면 기술 폭이 넓은 후보에게 이점은 주되,  
추천 순위를 혼자 결정하지는 못합니다.

---

# 7-2. 왜 최대 20점인가

기술 타입이 매우 많은 포켓몬이 과도하게 유리해지는 것을 막기 위해서입니다.

기술 폭은 "보조 판단 기준"이고,  
추천의 중심은 여전히 **약점 보완**과 **기본 전투력**입니다.

20점 제한은 의도적으로 약점 1개 보완 점수인 25점보다 낮습니다.

즉, 기술 타입이 아무리 많아도  
약점 보완 1개의 가치보다 크게 보지 않습니다.

---

# 8. duplicate_penalty

## 타입 중복 감점

후보 포켓몬의 타입 종류가 이미 현재 팀에 포함된 타입과 겹치면 감점합니다.

```text
duplicate_penalty = min(현재 팀과 겹치는 후보 타입 종류 수 * 20, 40)
```

예시:

```text
후보 타입이 팀 타입과 2종류 겹침
min(2 * 20, 40) = 40점 감점
```

이 감점은 6번째 포켓몬이 기존 팀과 다른 방어/공격 축을 제공하도록 유도하는 장치입니다.

---

# 8-1. 왜 20점을 감점하는가

팀 빌더의 목적은 단순히 강한 포켓몬을 추가하는 것이 아니라,  
현재 5마리 팀에 부족한 타입 축을 보완하는 것입니다.

따라서 후보 타입이 기존 팀과 겹치면 중복 타입 1종류당 20점을 감점합니다.

```text
1종류 중복 -> 20점 감점
2종류 중복 -> 40점 감점
3종류 이상 중복 -> 최대 40점으로 제한
```

20점은 기술 커버리지 최대치인 20점과 같은 규모입니다.  
즉, 기술 폭이 아무리 좋아도 타입 중복 1종류를 완전히 상쇄하기 어렵게 만든 값입니다.

---

# 8-2. 왜 최대 40점인가

타입 중복을 너무 약하게 감점하면 추천 결과가  
이미 팀에 많은 타입을 계속 반복하는 방향으로 쏠릴 수 있습니다.

반대로 감점을 무한정 크게 주면, 약점 보완을 잘하는 후보도 과하게 탈락합니다.

그래서 최대 감점을 40점으로 제한했습니다.

```text
약점 2개 보완 50점 - 타입 2종류 중복 40점 = 여전히 10점 이득
약점 1개 보완 25점 - 타입 2종류 중복 40점 = 중복 부담이 더 큼
```

즉, 중복 감점은 "중복 타입 금지"가 아니라  
약점 보완이 충분하지 않은 후보를 강하게 걸러내는 균형 장치입니다.

---

# 9. Graph Score 계산 예시 1

## 후보 A

| 조건 | 값 |
| --- | --- |
| 보완 약점 타입 | 바위, 얼음, 전기 |
| base_total | 600 |
| 배울 수 있는 기술 타입 수 | 12 |
| 팀과 겹치는 타입 종류 수 | 1 |

```text
defensive_score = 3 * 25 = 75
stat_score = min(600 / 140, 5) = 4.29
coverage_score = min(12 * 1.5, 20) = 18
duplicate_penalty = min(1 * 20, 40) = 20
graph_score = 75 + 4.29 + 18 - 20 = 77.29
```

---

# 10. Graph Score 계산 예시 2

## 후보 B

| 조건 | 값 |
| --- | --- |
| 보완 약점 타입 | 바위 |
| base_total | 680 |
| 배울 수 있는 기술 타입 수 | 16 |
| 팀과 겹치는 타입 종류 수 | 2 |

```text
defensive_score = 1 * 25 = 25
stat_score = min(680 / 140, 5) = 4.86
coverage_score = min(16 * 1.5, 20) = 20
duplicate_penalty = min(2 * 20, 40) = 40
graph_score = 25 + 4.86 + 20 - 40 = 9.86
```

후보 B는 능력치는 높지만, 팀 약점 보완이 적어서 후보 A보다 낮습니다.

---

# 11. Vector Score

Vector Score는 RAG에서 사용하는 보조 점수입니다.

후보 포켓몬의 이름, 타입, 보완 타입, 대표 기술명이  
검색된 문서 근거와 얼마나 잘 맞는지 확인합니다.

```text
vector_score = 검색 문서 상위 3개의 유사도 평균을 0~100 기준으로 정규화
```

문서 근거가 없거나 관련성이 낮으면 Vector Score는 낮게 나옵니다.

---

# 12. Hybrid Score

RAG 추천의 최종 순위는 Hybrid Score를 기준으로 정렬합니다.

현재 추천 가중치는 다음과 같습니다.

```text
Graph DB : Vector DB = 8 : 2
```

공식은 다음과 같습니다.

```text
normalized_graph_score = min(graph_score / 150 * 100, 100)
hybrid_score = normalized_graph_score * 0.8 + vector_score * 0.2
```

여기서 `graph_score`는 최대 150점으로 설계한 원본 점수이고,  
`normalized_graph_score`는 Vector Score와 같은 0~100 기준으로 맞춘 점수입니다.

## 왜 추천은 8:2인가

추천 순위는 실제 타입 상성과 팀 약점 보완이 가장 중요합니다.

따라서 Graph DB 계산 결과를 80%로 두고,  
Vector DB 근거는 설명과 보조 판단을 위해 20%만 반영합니다.

| 점수 | 역할 |
| --- | --- |
| graph_score | 타입 상성, 약점 보완, 능력치, 기술 폭을 이용한 핵심 추천 점수 |
| vector_score | 문서 근거와 후보 포켓몬 설명이 얼마나 잘 맞는지 보는 보조 점수 |

즉, 추천 순위는 Graph DB가 계산한 약점 보완 능력을 우선 신뢰합니다.

Vector Score는 후보의 기술 설명, 특징, 전략 문맥이 추천 이유와 잘 맞는지 확인하는 보조 역할입니다.  
그래서 Vector 점수가 높아도 Graph 점수가 낮으면 1순위가 되기 어렵습니다.

---

# 13. 분석/추천/해설 가중치를 다르게 둔 이유

팀 빌더에서는 Graph DB와 Vector DB를 항상 같은 비율로 섞지 않습니다.  
각 단계의 목적이 다르기 때문입니다.

| 단계 | Graph DB | Vector DB | 설계 의도 |
| --- | ---: | ---: | --- |
| 덱 분석 | 70% | 30% | 현재 팀의 약점, 저항, 기술 타입 커버리지는 타입 상성 계산이 중심입니다. |
| 포켓몬 추천 | 80% | 20% | 추천 순위는 6번째 포켓몬이 팀 약점을 실제로 보완하는지가 가장 중요합니다. |
| LLM 해설 | 60% | 40% | 답변은 계산 결과를 지키면서도 기술명, 역할, 전략 설명이 풍부해야 합니다. |

## 덱 분석을 7:3으로 둔 이유

덱 분석은 “현재 팀이 어떤 타입에 약한가”를 판단하는 기능입니다.  
이 판단은 타입 상성, 평균 데미지 배율, 기술 타입 분포처럼 Graph DB에서 계산한 값이 핵심입니다.

다만 사용자는 숫자만 보면 이해하기 어렵기 때문에 Vector DB 근거를 30% 반영합니다.  
Vector DB는 포켓몬 설명, 기술 설명, 전략 문맥을 보태서 분석 문장을 더 자연스럽게 만드는 역할을 합니다.

## 포켓몬 추천을 8:2로 둔 이유

추천은 실제 순위가 바뀌는 기능입니다.  
따라서 문서 설명보다 “팀 약점을 얼마나 막아주는가”가 더 큰 기준이어야 합니다.

예를 들어 어떤 후보가 문서상으로는 강한 포켓몬처럼 보여도,  
현재 팀이 약한 바위/얼음/전기 타입을 막지 못한다면 추천 우선순위가 높아지면 안 됩니다.

그래서 추천 단계에서는 Graph DB를 80%로 높이고,  
Vector DB는 동점 후보를 구분하거나 추천 이유를 보강하는 정도로 20%만 반영합니다.

## LLM 해설을 6:4로 둔 이유

LLM 해설은 순위 계산이 아니라 사용자가 이해할 수 있는 설명을 만드는 단계입니다.  
그래서 Graph DB 계산 결과를 60%로 중심에 두되, Vector DB 근거를 40%까지 사용합니다.

이렇게 하면 “왜 이 포켓몬이 추천됐는지”를 단순 점수 나열이 아니라  
기술 활용, 역할, 보완 타입, 전투 상황 중심으로 설명할 수 있습니다.

# 14. Hybrid Score 계산 예시 1

## 후보 A

```text
graph_score = 77.29
normalized_graph_score = 77.29 / 150 * 100 = 51.53
vector_score = 76
```

```text
hybrid_score = 51.53 * 0.8 + 76 * 0.2
             = 41.22 + 15.2
             = 56.42
```

Graph 점수가 높고 Vector 근거도 충분해서 최종 점수가 높습니다.

# 15. Hybrid Score 계산 예시 2

## 후보 B

```text
graph_score = 9.86
normalized_graph_score = 9.86 / 150 * 100 = 6.57
vector_score = 20
```

```text
hybrid_score = 6.57 * 0.8 + 20 * 0.2
             = 5.26 + 4
             = 9.26
```

능력치는 높지만 약점 보완이 적고 타입 중복 감점이 커서 최종 점수가 낮아집니다.

# 16. 최종 정리

팀 빌더 추천 점수는 단순히 강한 포켓몬을 고르는 방식이 아닙니다.

가장 중요한 기준은 다음 세 가지입니다.

1. 현재 팀의 약점을 얼마나 잘 보완하는가
2. 기존 팀과 타입이 과하게 겹치지 않는가
3. 기본 능력치, 기술 폭, RAG 문서 근거가 보조적으로 뒷받침되는가

따라서 추천 결과는  
**강한 포켓몬**보다 **현재 팀에 필요한 포켓몬**을 우선합니다.
